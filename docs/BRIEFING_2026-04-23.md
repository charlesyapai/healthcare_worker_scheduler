## 2. What each phase actually gives a reviewer

### Phase 1 — feasibility receipt + fairness

Every solve now carries a `SolverSelfCheck`: the hard-constraint
validator re-checks the solver's output and reports pass/fail per
rule. If the CP-SAT model and `api/validator.py` ever disagree (the
only scenario where this would go red), the backend logs a WARNING
and the UI turns the badge red with the failing rule list. The green
path is the "feasibility receipt" [VALIDATION_PLAN §1.1](VALIDATION_PLAN.md)
asked for.

On `/roster`, the new **Fairness & bias audit** card surfaces, per
tier:

- FTE-normalised weighted workload → Range, **Gini** (our
  convention), **CV** (the NRP literature standard), std, mean
- Per-individual delta-from-median with outlier flagging (top/bottom
  quartile)
- Day-of-week load distribution (stacked bar chart)
- Consultant subspec parity table

Formulae in [`docs/RESEARCH_METRICS.md §4`](RESEARCH_METRICS.md).
Hand-computed against a Gini = 10/60 fixture so a regression breaks
the test, not the UI.

### Phase 2 — comparative benchmarking

`scheduler/baselines.py` adds two heuristics that return
`SolveResult`-compatible objects:

- `greedy_baseline` — weekend H8 → weekday on-call → station coverage,
  lowest-load-first. Sanity floor per [INDUSTRY_CONTEXT §6](INDUSTRY_CONTEXT.md).
- `random_repair_baseline` — random + H1/H3/H10 repair. Weak by
  design; the test `test_random_repair_fails_self_check` locks in
  that it should NOT satisfy H4/H5/H8 (so CP-SAT's lift is
  measurable).

`/lab/benchmark` runs a cross-product of (solvers × seeds) on the
current state and shows the industry-reliability metrics the plan
asked for:

- **Feasibility rate per method** (§7.3) — fraction of runs with
  green self-check.
- **Mean objective + mean coverage shortfall** per method (§5.1b).
- **Quality ratio Q = Z_baseline / Z_ours** (§7.1) — populated only
  when both solvers return an objective (currently CP-SAT only;
  see §4 below).

Coverage shortfall + over-coverage are first-class metrics via
`api/metrics/coverage.py` + `POST /api/metrics/coverage`.

### Phase 3 — reproducibility

Bundle export: `GET /api/lab/runs/{batch_id}/bundle.zip` returns a
ZIP with every artefact VALIDATION_PLAN §1.3 specifies
(`state.yaml` + `run_config.json` + `results.json` + `git_sha.txt` +
`requirements.txt` + `README.md`). The session YAML is snapshotted
at run time so later state edits don't poison the bundle.

`scripts/replay_bundle.py` unpacks the ZIP, re-runs every cell, and
diffs against the recorded summary. A clean run prints
`OK — all N runs reproduce.`; a divergent run prints each mismatched
field. The deterministic path (num_workers=1 + fixed seed) is locked
into `test_cpsat_deterministic_with_fixed_seed_single_worker`.

`scheduler.solve()` gained six additive CP-SAT kwargs
(`random_seed`, `search_branching`, `linearization_level`,
`cp_model_presolve`, `optimize_with_core`, `use_lns_only`) with
defaults matching OR-Tools defaults so every existing call site is
unaffected. All six are exposed on the benchmark UI's **Advanced
CP-SAT knobs** panel.

Git SHA flows through `GET /api/health`, `git_sha.txt`, and the
bundle README.

### Phase 4 — deeper research surfaces

- `/lab/sweep` — sweep one CP-SAT parameter across a list of values;
  reports ΔZ_θ (objective delta) + ΔT_θ (wall-time delta) and a
  per-value summary table + line chart. Pure client-side composition
  over the existing `/api/lab/run` endpoint.
- `/lab/fairness` — pick any (batch × solver × seed) and see the
  full fairness drill-down plus a coverage-gap audit (top 10
  (date, station, session) tuples with shortfall). Renders the
  `SingleRunDetail.fairness` payload already stored at batch time
  — no extra backend round-trip.

### Phase 5 — docs

- [`docs/HOW_TO_REPRODUCE.md`](HOW_TO_REPRODUCE.md) — step-by-step
  replay walkthrough, determinism guarantees, what replay does NOT
  verify.
- [`docs/CITING.md`](CITING.md) — BibTeX stubs (solver + OR-Tools) +
  methodological citation pointers.
- README gains a **Research usage** section that explains what
  `/lab` exposes and links to the above.

## 3. Where the original goal stands

[`VALIDATION_PLAN.md §4`](VALIDATION_PLAN.md) defines success as:

> A reviewer reads our submission. They:
> 1. Clone the repo at the cited git SHA.
> 2. Pull the bundled benchmark instance.
> 3. Run `pnpm install && python -m pytest tests/` → green.
> 4. Run `python scripts/run_benchmark.py instance=NSPLib_001 seed=0..29`
>    → reproduces our reported mean ± std within rounding.
> 5. Open the Lab tab, run the same instance with our solver and the
>    greedy baseline, see the gap we claim.
> 6. Walk away convinced the system works as we say it does.

Going criterion by criterion:

| # | Criterion | Status |
|---|---|---|
| 1 | Clone at cited git SHA | ✅ SHA exposed on `/api/health`, embedded in every bundle. |
| 2 | Pull the bundled benchmark instance | ⚠ Our three *scenarios* are bundled; a public benchmark (NSPLib / INRC-II) is not. See §4. |
| 3 | `pytest tests/` → green | ✅ 52/52 pass in ~4 min. |
| 4 | `python scripts/run_benchmark.py ...` reproduces mean ± std | ✅ equivalent via `replay_bundle.py` + `/lab/sweep`. The "run a benchmark" script the plan hypothesised maps onto our bundle replay + the sweep tab. |
| 5 | Open Lab, run same instance, see the gap vs greedy | ✅ `/lab/benchmark` is exactly this surface. |
| 6 | Walk away convinced | ⚠ Conditional — see §4. |

**In short:** we can clear criteria 1, 3, 4, and 5 today. Criteria 2
and 6 hinge on the gaps below.

## 4. Known gaps (things NOT shipped)

These were visible in the plan but out of scope for this pass. Listed
with the reason:

### 4.1 Public-benchmark adapter (NSPLib / Curtois / INRC-II)

The plan asks us to adapt "at least one" NSPLib/Curtois instance into
our YAML format so a reviewer can compare against published NRP
numbers. We did not do this — building a translator is a ~1-day
effort on its own and the deliverable is primarily about comparable
*numbers*, not UX. Phase 5 in the plan explicitly lists this as its
terminal step ("Adapt one NSPLib instance"); we have not reached it.

Impact: **Criterion 2 in §3 is unmet.** The Lab today benchmarks
against our own three scenarios, which are scenarios *we shipped*,
not benchmarks a reviewer can cross-reference with a published
paper. The `quality_ratios` metric is computed but will only ever
compare CP-SAT against greedy/random on our own scenarios until
this lands.

Follow-up: `lib/objective_translator.py` + at least one NSPLib
instance dropped into `configs/scenarios/`. Tests exist for the
machinery; only the translator is missing.

### 4.2 PuLP + CBC MILP baseline

`docs/INDUSTRY_CONTEXT.md §6` flags PuLP+CBC MILP as the **#1
priority baseline** in NRP papers. We have greedy + random_repair,
both of which return `objective=None` because they don't optimise
our weighted objective. This means `quality_ratios` in the Lab's
reliability banner will stay empty until there's a MILP baseline
that *does* return an objective comparable with CP-SAT's.

Impact: The Q metric is implemented and tested but there is no
situation today in which it lights up. That's honest — the code is
ready for the MILP baseline.

Follow-up: `scheduler/baselines.py` gains a `milp_baseline(inst)`
using PuLP's CBC backend. Requires adding `pulp` to
`requirements.txt` and ~200 lines encoding H1–H15. Half a day.

### 4.3 Regulatory-conformance module

`docs/INDUSTRY_CONTEXT.md §5` asks for at least one jurisdiction's
statutory rules (UK WTD, ACGME, or California AB394) as a pluggable
hard-constraint layer. Not started. The self-check currently
verifies H1–H15 + weekday on-call; it does NOT verify, e.g., "no
doctor worked more than 48h in any 7-day window". A mathematically
feasible roster from our solver is not guaranteed to be statutorily
compliant.

Impact: A paper that claims "suitable for NHS deployment" would be
overclaiming. CITING.md already notes this caveat.

Follow-up: `api/compliance/uk_wtd.py` with the 6 rules from
`docs/INDUSTRY_CONTEXT.md §5` + a toggle on the constraints config
+ a second self-check that runs regulatory rules. ~1 day.

### 4.4 `/lab/scaling` sub-tab

The plan's LAB_TAB_SPEC §5 specifies a scaling tab (log-log chart of
solve time vs problem size). Deferred — `/lab/sweep` gives 80% of
the same signal when you sweep `time_limit_s` or `num_workers` on a
fixed instance. Scaling would use `make_synthetic` to generate a
grid of instance sizes and time each.

Follow-up: ~half a day of UI + a synthetic-instance endpoint.

### 4.5 Background-job path for long batches

Currently `POST /api/lab/run` is synchronous. `docs/LAB_TAB_SPEC.md §9.1`
flagged this as a Phase-3 polish; we kept the synchronous path and
added a "don't close the tab" warning in the UI. For batches over ~2
minutes, this is user-hostile. Still, the synchronous model is
correct for the 95% use case (≤30 runs × ≤30 s budget), so the
deferral is defensible.

## 7. What a reviewer sees if they open the Space today

1. **Dashboard** — loads a scenario.
2. **Solve** — runs CP-SAT with live WS streaming; the green
   **Solver self-check** badge appears when finished.
3. **Roster** — colour-coded heatmap, workload, score breakdown, and
   the new **Fairness & bias audit** card.
4. **Lab → Benchmark** — picks solvers + seeds, presses Run. Sees
   per-solver reliability cards (feasibility rate, mean objective,
   mean shortfall), quality-ratio chips, per-run table with
   self-check + coverage columns. Clicks **Download bundle** →
   gets the ZIP.
5. **Lab → Sweep** — picks a CP-SAT parameter, a value list, seeds.
   Sees ΔZ / ΔT per value and a line chart of objectives.
6. **Lab → Fairness** — picks any recorded run, sees the full
   fairness drill-down + coverage-gap audit.
7. Downloads `bundle.zip`, follows `docs/HOW_TO_REPRODUCE.md`,
   and `replay_bundle.py` confirms reproduction.

That is the full chain of evidence VALIDATION_PLAN asked for,
modulo the external-benchmark and MILP items flagged in §4.
